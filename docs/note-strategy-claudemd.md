今回から3部作で、「CLAUDE.mdになにを書くべきか」について、ポエムを書いてみます。

第一部：みんな間違った場所にいる

CLAUDE.mdについて調べると、どのブログを読んでも、どのドキュメントを見ても、同じようなことが書いてあります。

「Claudeに守らせたいルールを書く場所です」 「コーディング規約や、やってはいけないことを記述しましょう」 「使いながら育てていくことが大切です」

悪意があるわけではありません。みんな誠実に、自分の経験から書いています。でも構造的に、全員が同じ場所で間違えています。

/initが生み出すもの

Claude Codeを使い始めると、まず/initコマンドを勧められます。プロジェクトのディレクトリで実行すると、CLAUDE.mdが自動生成されます。すると、コードベースをスキャンして、使われている技術スタック、ディレクトリ構造、検出されたコマンドが整然と書き出されます。

これが、最初の罠です。

/initが生成するCLAUDE.mdには、「このプロジェクトが何のためにあるか」が一行も書かれていません。AIがスキャンできるものしか、AIは書けないからです。プロジェクトの存在理由は、コードの中には存在しないでしょう。

そして多くの人は、この自動生成されたファイルをベースに、ルールを追記し始めます。前述した引用のようにです。

典型的な管理型CLAUDE.mdの解剖

実際によく見かけるCLAUDE.mdは、こんな感じです。

## 基本方針
- 回答・説明は日本語で簡潔に
- 20行を超える修正は確認する
- .envファイルの中身は絶対に参照しないこと

## 開発ルール
- コーディングスタイルはPEP 8に従う
- 既存の設計・命名・フォーマットを優先する

## テストルール
- 修正したコードにpylintで構文チェックを実施する
- テストコードを作成しpytestでユニットテストを実施する

## その他ルール
- 勝手にcommit/pushしない

これを書いた人はおそらく優秀です。必要だと思うことを、きちんと考えて書いています。でも、このファイルをよく読むと、あることに気づきます。

.envファイルを参照しないこと——これは過去に何かあったのでしょう。

勝手にcommit/pushしない——これも、過去に何かあったのでしょう。

これらは全部、過去の失敗が書いてあるようです。

実は、これが「CLAUDE.mdを育てる」の正体です。

失敗する、追記する。また失敗する、また追記する。その繰り返しの中で、ファイルは肥大化していきます。終わりがありません。なぜなら、Claudeが次に何で失敗するかは、事前には予測できないからです。

「育てる」という言葉の罠

「CLAUDE.mdを育てる」という表現は、ネット上いつの間にか広まったものです。Anthropicの公式ドキュメントには、この言葉は存在しません。

公式が使う言葉は「persistent instructions（持続的な指示）」であり、ClaudeはCLAUDE.mdを「enforced configurationではなく、contextとして扱う」と言っています。

Claude treats them as context, not enforced configuration

それは、CLAUDE.mdの「管理型」の構造が、必然的に「育てる」という行為を生み出すからです。

追記するたびに愛着が生まれる

愛着があるから「育てている」と感じる。

でも本当は、失敗や傷の記録が増えているだけであり、ここに、根本的な問いが生まれます。

それは、CLAUDE.mdとは何のためのファイルなのか。ルールを書く場所なのか。コンテキストを注入する場所なのか。それとも、もっと別の何かなのか。

傷の記録か、意志の宣言か

「このプロジェクトで何があったか」

前者はCLAUDE.mdを「管理型」として扱ってるパターンです。失敗の蓄積として育てられたファイルは、プロジェクトの歴史の傷跡を記録しています。

一方で、後者は「目的型」として扱っているパターンです。プロジェクトの存在理由と判断軸が、数行で書かれているでしょう。

CLAUDE.md

中小企業の経理担当者が、月次決算を一人でこなせるようにするためのツールである。
ユーザーは会計の専門家ではない。難しさより、安心感を優先する。

判断に迷ったときは「経理初心者が一人で使えるか」を基準にする。
技術的な正しさより、ユーザーが途中で止まらないことを優先する。

このコードを触るのは3名の小さなチームで、全員がフルスタックに動く。

管理型のCLAUDE.mdを書いている人に、悪意はありません。公式ドキュメントも、/initも、ブログも、Xのポストも、管理型にしかなりようがなかったでしょうから。

第二部：目的は育てない、書く

前回お伝えした、Anthropic公式による「CLAUDE.mdをコンテキストとして扱う」という説明を、多くの人は「ルールを書いておく場所」と解釈しています。

しかし、よく考えてみると、contextとは、文脈であり、背景であり、状況です。「CLAUDE.mdとはenforced configuration（強制された設定）ではない」、という言葉に、CLAUDE.mdの本当の意味が宿っているのではないでしょうか。

Claudeは、ルールに従うのではなく、文脈を読んで判断する。ならば、CLAUDE.mdに書くべきは、判断の根拠となる「文脈」であって、判断の結果を縛る「ルール」ではないはずです。

Why-What-Howで読み解くCLAUDE.md

ビジネスの文脈でよく使われる、Why-What-Howの3層構造があります。

Why：なぜこれをやるのか。目的、存在理由。

What：何をやるのか。戦略、方針。

How：どうやるのか。戦術、手段、ルール。

「管理型」のCLAUDE.mdは、すべてHowのレイヤーで書かれています。

「PEP 8に従う」

「pushしない」

「確認する」

これらはすべて「How」です。

Howだけが書かれたCLAUDE.mdで何が起きるか。

Claudeは、目的を知らないまま、ルールの網の目をくぐるように動きます。想定外の状況でルールが衝突したとき、どちらを優先すべきかわかりません。ルールに書いていないことは、自己判断になる。その自己判断が、また失敗を生み、そして、われわれはまた追記する。

Whyがないから、Howが増え続ける。

ケニー節

これが、CLAUDE.mdが肥大化するメカニズムと言えるでしょう。ばあいによっては「AI疲れ」とも言えるかもしれません。

目的型と管理型、一目でわかる違い

ケニーともなると、CLAUDE.mdを見た瞬間に、「管理型」か「目的型」なのかがわかります。

管理型のCLAUDE.md箇条書きが続く。禁止・命令形が多い。読むと、不安が伝わってきます。

- 必ずテストを書くこと
- コミットメッセージは英語で
- xxx.tsは編集禁止
- 必ずユーザーに確認してから進めること

目的型のCLAUDE.md

このプロジェクトは○○のためにある。
ユーザーは△△という文脈でこれを使う。
判断に迷ったときは□□を優先する。

短い。動詞より名詞と関係性で書かれている。読むと、意志が伝わってきます。

ディレクトリ構造にも違いが出ます。

管理型のプロジェクトは、

project/
├── CLAUDE.md                    # 全体ルール集
└── .claude/
    └── rules/
        ├── coding.md            # コーディング規約
        ├── git.md               # Gitルール
        ├── testing.md           # テストルール
        ├── security.md          # セキュリティ禁止事項
        └── communication.md     # 確認・報告ルール

と、増殖の跡が見えます。カテゴリが増えるたびにファイルが増えます。

目的型のプロジェクトは、CLAUDE.mdの階層が「目的の粒度」を表しています。

project/
├── CLAUDE.md                    # このプロダクトは何のためにあるか
└── src/
    ├── CLAUDE.md                # このドメインが解く問題
    ├── api/
    │   └── CLAUDE.md            # このAPIが担う責務と判断軸
    └── billing/
        └── CLAUDE.md            # 決済まわりの文脈と優先順位

階層を降りるほど、目的が具体化していく。ルールではなく、粒度の違う目的が積み重なっています。

会話のパターンにも違いが出ます。

管理型のプロジェクトでは、Claudeがルールの確認を求めてきます。

「このファイルは編集してよかったですか？」

人間が修正して、またルールに追記する。

一方で、目的型のプロジェクトでは、Claudeが目的から逆算して提案してきます。

「この実装、プロジェクトの目的からすると、別の方向の方が整合しませんか？」

 前者はツールの動作で、後者はエージェントの動作です。管理型は「何をしてはいけないか」の地図。 目的型は「なぜここにいるか」の地図という感じです。

ルールは育てる。目的は書く。

「CLAUDE.mdを育てる」という話に戻ります。

ルールは育てる。失敗するたびに追記して、時間をかけて整備していく。それは正直な営みです。

しかし、目的は育てるものではありません。

組織のミッションを「育てる」とは言いいません。目的は、考え抜いて書くもの。一度書いたら、変わらない核として機能します。変わるのは実装であって、目的ではありません。

なので、目的型のCLAUDE.mdは最初から短くなります。そして、むしろ削っていくものです。「これも言っておかなければ、あれも伝えなければ」という管理型の不安が、CLAUDE.mdを肥大化させます。

目的が明確なら、Claudeは目的から演繹して動きます。ルールに書いていないことも、判断できます。`.envを見るな`と書かなくても、「このプロジェクトのセキュリティ要件と、このコードが動く環境を理解していれば」Claudeは自分でそう判断します。

ルールで縛るか、目的を渡して信頼するか。

それはAIとの関係の問題である前に、委ねることへの哲学の問題かもしれません。

第三部：contextは会社の資産になる

ここまで、CLAUDE.mdの「管理型」と「目的型」の違いを見てきました。

CLAUDE.mdには目的を書け、ルールを書くな。

ケニー節

しかしながら、目的だけ書けば終わりという話ではありません。

目的は育てない。でもcontextは積み上げる。

ここで、重要な区別をしなければなりません。

目的は育てない。でも、contextは積み上げる。

この二つは、レイヤーが違います。

目的（Why）は、変わらない核です。プロジェクトの存在理由は、書き直すものではありません。でも、目的の下にあるcontext（What）は、蓄積するものです。

そして、積み上げるべきは、目的から演繹された文脈です。「失敗の記録」ではなく、「判断の蓄積」を積み上げていくのです。

資産になるcontextとは何か

資産になるcontextを、具体的に見ていきましょう。

ドメイン知識の蓄積

医療系スタートアップなら「この国の薬事規制では、XXはYYと解釈する」という業界固有の知識などです。新しいエンジニアが入っても、AIエージェントのモデルが変わっても、このcontextは引き継がれるべきです。コードベースより長く生き残るかもしれません。

2. 意思決定の文脈

「うちはBtoBで、顧客の社内稟議を通すことがボトルネックになっている」という商流の理解など。これをcontextとして持っているエージェントは、機能開発の優先順位を間違えません。「便利な機能」より「稟議書に書きやすい機能」を選ぶ判断ができるでしょう。

3. 失敗の蒸留

過去の失敗をルールとして書くのが「管理型」でした。でも、なぜ失敗したかを目的に照らして蒸留し直したcontextは資産になりえます。

管理型：「XXXは禁止」 

目的型：「XXXを試みたが、私たちの目的には合わなかった。ユーザーがYYYという文脈で使うため、ZZZの方向が整合する」

前者は制約であり、後者は判断軸です。

4. チームの優先順位

「私たちはスピードより一貫性を優先する」「完璧な設計より、動くものを早く届けることを選ぶ」という意思決定の原則です。これはルールではなく文化ですね。AIエージェントがこれを持っていると、都度確認せずに動けます。

ハーネスエンジニアリングの時代における、設計者の仕事

少し、大きな話をします。

「AIエージェント」というエネルギーを持つ存在に、人間がやるべきことは何でしょうか。

ハーネス、つまり手綱を引くことではありません。それは「どこへ向かうかを決めること」に他ならないでしょう。

管理型のCLAUDE.mdは、手綱を引こうとしている。ルールで縛り、禁止事項を並べ、逸脱を防ごうとする。でも、エネルギーを縛るほど、エージェントはエージェントではなくなり、ツール化します。

目的型のCLAUDE.mdは、方向を示し、Whyを渡す。判断軸を委ねる。するとエージェントは、目的から演繹して動く。想定外の状況でも、目的に立ち返って判断できる。ルールの網の目をくぐるのではなく、目的の方向に向かって動きだします。

これが、ハーネスエンジニアリングの本質です。

最後に:CLAUDE.mdを書くという行為の意味

3回にわたってお届けしてきた、「CLAUDE.mdに何を書くべきか」ですが、最後に、最も重要なことをお伝えします。

CLAUDE.mdに目的を書こうとすると、必ず壁にぶつかる。

ケニー節

CLAUDE.mdには、このプロジェクトは何のためにあるのか。誰のために、何を解決しようとしているのか。判断に迷ったとき、何を優先すべきなのか。を書くべきです。

しかし、これを数行で書けないとき、問題はCLAUDE.mdにあるのではなく、まだ考え抜いていないということを知りましょう。

AIエージェントに目的を渡せない人は、自分でも目的を持っていない

CLAUDE.mdを書くという行為は、実はAIのための作業ではありません。自分のプロジェクトのWhyを、強制的に言語化させる行為に他なりません。AIへの指示という実用的な文脈が、思考の鏡として機能しているだけです。

ルールを書くのはかんたんです。失敗を残していけばいいからです。しかし、 目的を書くのは難しいです。考え抜かなければならないからです。

その難しさから逃げて、ルールを書き続けることを、誰もが「CLAUDE.mdを育てる」と呼んできたのかもしれません。

本当に育てるべきは、CLAUDE.mdではない。目的を言語化する力だ。

ケニー節

AIと働く時代の設計者とは、コードを書く人ではなく、目的を書ける人のことなのかもしれません。
